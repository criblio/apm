## Follow-up items from eval

### Must fix (improve detection)
1. **Add latency anomaly alerting to the evaluator** — adHighCpu, adManualGc, kafkaQueueProblems all fail because the alert evaluator only tracks error_rate/traffic_drop/silent. Need to join against `criblapm_op_baselines` lookup in the evaluator query and emit latency_anomaly alerts.
2. **Don't add error_rate alert checks to non-error scenarios** — loadGeneratorFloodHomepage is a rate spike, not an error scenario. The eval checks should match the scenario's signal type, not blindly check for error_rate alerts on every scenario.
3. **svcDetailAlertBadge locator** — fails on most scenarios. The Playwright locator `h1:has-text("svc") span:text-matches("firing|pending")` may need adjustment for the actual DOM structure, or the alert state hasn't reached firing within the check timeout for weaker signals.

### Should fix (improve reliability)
4. **Inter-scenario bleed** — cartFailure scored 0.25 despite being a strong error signal. Running 13 scenarios sequentially over 6+ hours accumulates state. Consider resetting alert state (clearing the lookup) between scenarios.
5. **emailMemoryLeak timing** — the OOM crash produces a brief error burst then recovery. The 7-minute telemetry wait may miss the burst. Increase telemetryWaitMs for slow-onset scenarios.
6. **failedReadinessProbe** — errors propagate indirectly through k8s. May need longer wait or different surface checks (log-based rather than error-rate).

### Nice to have
7. **Per-scenario signal type** — add a `signalType` field to ScenarioDeclaration so the eval can generate alert checks appropriate to the scenario (error_rate checks for error scenarios, latency checks for latency scenarios, traffic_drop for traffic scenarios).
8. **Alert state reset between scenarios** — seed the `criblapm_alert_states` lookup with a clean state before each scenario to prevent carryover.

## Cribl KQL bugs discovered

1. **`any()` function** — `Unknown Kusto scalar function 'any'` on some Cribl Search versions. Fixed by using `max()` instead.
2. **`summarize → summarize max(iff(...))`** — crashes with "Unknown error" on real data (works on 4 synthetic rows, fails on 36 real rows from a prior summarize). Worked around by splitting into separate scheduled searches + lookups. Bug report pending.
